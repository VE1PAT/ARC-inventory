Amateur Radio Club Inventory System

Proposal for club comment — web-based asset tracking, loans, and accountability

Draft for discussion · Please review and comment by email

1. Project description (as proposed)

We need a web-based inventory management system for the club, secured with individual logins using each operator’s callsign and their own password.

The purpose of the app is to track all club assets, including:

  • Physical hardware
  • Software
  • Books and other materials
  • Other club property as needed

Club members can sign out items on loan. Loans need to be verified by another club member for both loan-out and loan return. There should be a ledger tracking dates, callsigns, and items.

The system will run on Linux in a Proxmox VM. Storage will be MySQL, but accountability is of the utmost importance.

Some items being tracked are donations from silent keys (deceased operators). These items may later be sold at local flea markets or to club members. When that happens, we do not need to track money — only inventory status.

The webpage must be fully usable from a cell phone and from a laptop/desktop. The interface must be as simple as possible, with a keyword search that returns results from multiple fields.

Open question from the proposer: maybe something that already does this exists — that possibility is addressed in the recommendation below.

2. Recommendation summary

Build a small custom app rather than forcing a general IT asset system into club workflow. Off-the-shelf tools get roughly 70% of the way; the dual-member verification for both loan-out and return is the part that usually does not fit cleanly.

The killer feature is two-person loan/return verification plus an immutable ledger — especially important for donated estate gear. That is worth a focused club app more than adapting a corporate IT tool.

3. What already exists

Option Fit Gap for our club
Snipe-IT Strong asset tracking, check-in/out, self-host on Linux Built for IT checkout by admins — not “two members witness every loan/return”
Asset Lending Manager (WordPress) Clubs/associations, loan requests, history Approve-by-owner model, not dual verification; WordPress overhead
Shelf.nu Modern booking/custody tracking More studio/ops booking than club trustee workflow

None of these cleanly model: member A loans to member B, member C verifies; later B returns, member D verifies — with an immutable ledger of who did what when.

4. Requirements, ranked by importance

  1. Dual verification — design centerpiece. Treat it like a two-person control: borrower + verifier for out; returner + verifier for in. Verifier must not be the borrower.
  2. Append-only ledger — every status change is a new row (who, when, item, action, verifier). Never edit history away. That supports accountability for silent-key donations and sales.
  3. Statuses, not money — examples: available, on_loan, maintenance, sold, disposed, reserved. A sale is a status change plus a note (“sold at flea / to W1ABC”), with no cash fields.
  4. Callsign login — username = callsign (normalized uppercase). Simple roles: member, inventory_officer, admin.
  5. Search — one keyword box across callsign, description, manufacturer, model, serial, location, notes, donor, tags.
  6. Mobile-first UI — big buttons, few screens: Search → Item → Loan / Return → Confirm with verifier.

5. Data storage recommendation

Prefer MySQL / MariaDB over JSON files as the system of record. JSON files are fine for a prototype; for accountability, a database is better because it supports:

  • Concurrent loans without file corruption
  • Proper backups (mysqldump + Proxmox snapshots)
  • Easy “who changed what” queries
  • Foreign keys so ledger rows cannot be orphaned

JSON export of the ledger can still be useful as an audit/archive copy.

6. Suggested simple design

Items

Type (radio, antenna, book, software, cable, …), description, serial/ID, location, condition, donor/source, status, notes, optional photo.

Members

Callsign, password hash, name, active flag, role.

Ledger events (immutable)

Event types such as loan_out, loan_return, status_change, created, edited_metadata — each with actor callsign, verifier callsign (when required), timestamp, item ID, and notes.

Loan flow on phone

  1. Find item → choose Loan
  2. Enter borrower callsign (or self)
  3. Enter verifier callsign + password (or PIN)
  4. Confirm → item marked on_loan, ledger row written

Same pattern for return. Officers can mark sold / disposed with a note, and optionally a second confirmation for silent-key gear.

7. Technical fit for our Proxmox VM

  • Linux guest on Proxmox
  • MariaDB + a maintainable web stack (e.g. PHP/Laravel, Python/Django or FastAPI, or Node)
  • Reverse proxy (Caddy or Nginx) with HTTPS
  • Daily database dump + Proxmox VM backups

If the club already runs WordPress, Asset Lending Manager is worth a short trial — but dual-verify would likely still need custom work.

8. What to avoid

  • Spreadsheets as the system of record
  • Tracking money (already ruled out — keep it that way)
  • Overbuilding early: barcodes/QR can come later; start with search + clear item IDs
  • Allowing anyone to edit past ledger rows

9. Bottom line

The idea is sound and appropriately scoped. Dual verification and an immutable ledger make this a club-specific tool more than a generic asset manager.
